프로젝트 개요

결제 로직은 모놀리식, 환불과 정산은 어드민 서버에 흩어져 있어 PG사 추가나 변경 시 양쪽을 모두 수정해야 하는 구조. 결제 도메인 전체를 별도 결제 서비스로 분리

팀 구성 및 역할

  • 3명 백엔드 팀으로 진행
  • 주로 환불 도메인 전반과 PG 연동, Java 엔티티 모델링 작업을 진행함

주요 업무

  • Node.js 모놀리식의 모호한 결제 도메인 개념을 정리해 Java 엔티티와 값 객체로 재모델링
  • 모놀리식, 백오피스에 존재하는 결제와 환불 로직을 정리해 결제 시스템에 이관 및 연동
  • 나이스페이먼츠, 카카오페이먼츠, 토스페이먼츠 PG 연동(결제, 환불) 작업
  • 비동기 분리 작업

문제 해결 과정

  • 두 DB 동시 갱신 정합성
    • 대안 비교: 평균 50TPS 규모의 단일 인스턴스라 2PC와 Saga 패턴은 복잡도 대비 이득이 없다고 판단. 결제 → 레거시 순서로 커밋을 설계하면 레거시 DB 장애가 아닌 한 정합성이 깨지지 않아, Saga 패턴은 자주 발생하지 않을 상황에 대한 오버엔지니어링으로 배제
    • 2PC의 운영, 성능 부담 대신 ChainedTransactionManager로 두 DB를 순차 커밋
    • 잔여 리스크인 레거시 커밋 실패만 Slack 알림 + 복구 쿼리로 대응
  • 정산 비동기 분리
    • 결제 완료 시점에 SQS로 결제 이벤트 발행, 정산 서버 비동기 컨슘
    • 실패는 DLQ 재시도와 청구서 ID 기반 멱등성 키로 중복 정산 방지
  • 테스트로 도메인 정확성 보장
    • 단위와 통합 테스트 300개 이상 작성, 커버리지 95% 유지

기여 성과

  • 정산 및 로깅 처리 비동기화로 응답 속도 2.1s → 0.4s
  • 모놀리식, 백오피스 서버 무중단 분리 이관

지도 영역 조회 최적화

Full Scan 장애
처리 속도 6s → 0.4s (15배)

Java 17, Spring Boot, Redis, MySQL 5.7

프로젝트 개요

평균 250RPS, 피크 400RPS가 몰리는 모놀리식 메인 API에서 Full Scan 쿼리가 RDS CPU 사용량을 끌어올려, 같은 DB를 쓰는 다른 쿼리까지 느려지는 전체 장애로 확산. 캐싱 전략을 단계적으로 진화시키며 근본 원인까지 해결.

팀 구성 및 역할

  • 단독으로 원인 분석, 캐싱 전략 수립, 점진적 마이그레이션 담당

주요 업무

  • 운영 구역 폴리곤, 가맹점, 킥보드 위치 데이터를 매 화면 진입마다 응답하던 API 최적화

문제 해결 과정

  • 1단계: 로컬 캐싱
    • 변경 빈도 낮은 지역 데이터에 WAS 로컬 캐싱 적용으로 6s → 0.4s
    • 주차 규제 변경이 잦아지며 WAS 증설 시 인스턴스별 캐시 불일치 발생
  • 2단계: Redis 글로벌 캐싱
    • WAS 간 일관성 확보로 캐시 불일치 해소
    • 가맹점 증가로 응답 크기가 커지며 프론트 렌더링 지연 발생
  • 3단계: 공간 쿼리
    • MySQL Geometry 함수로 사용자 인접 지역만 조회하도록 변경
    • 응답 크기 자체를 축소해 렌더링 지연 해소

기여 성과

  • 처리 속도 6s → 0.4s (15배 개선)
  • RDS Full Scan으로 인한 장애 재발 차단

권역 이탈 반납 자동 격리

구역 밖에 버려진 킥보드를
다음 유저가 못 빌리도록 자동 잠금

Node.js, Express, MySQL Spatial Index, Read Replica

프로젝트 개요

킥보드가 운영 구역 밖에 반납되면 자동 잠금해 다음 유저가 빌리지 못하게 격리. 현장 직원이 정상 위치로 옮기면 자동 해제.

팀 구성 및 역할

  • 단독으로 도메인 로직, 공간 쿼리, 4개 반납 경로 통합 담당
  • 운영팀과 허용 반경 정책 협의 / 어드민과 CX 팀과 시각화 UX 협의

주요 업무

  • 반납 시점 GPS가 운영 구역 안인지 판별 후 구역 밖이면 킥보드 자동 잠금
  • 유저, 서버, 직원 종료 등 4개 반납 경로 정책을 공통 레이어로 일원화
  • 어드민과 CX 화면에 반납 위치 마커와 구역 폴리곤 시각화

문제 해결 과정

  • GPS 소스 선택
    • 사용자 앱 GPS는 실내나 터널에서 100m 이상 튀어 정상 유저 오탐 발생
    • 킥보드 단말기 GPS 1순위에 마지막 위치 보조 신호 OR 조합으로 오탐 차단
  • 경계선 vs 여유 버퍼
    • 경계선 정확 판별은 GPS 오차 1~2m로 정상 유저까지 잠김
    • 경계선과 여유 폭 N m 버퍼 폴리곤을 별도 컬럼으로 운영, 판별은 buffer 기준
    • ST_CONTAINS와 MySQL Spatial Index로 구역 수 증가에도 O(log n) 보장
  • 운영 트랜잭션 부하 분리
    • 공간 판별 쿼리는 Read Replica로 분리, Write DB는 상태 갱신만 처리

기여 성과

  • 운영팀 수동 모니터링이 반납 즉시 자동 격리로 대체
  • 이후 친구공유나 예약 반납 같은 신규 경로에 동일 정책 자동 적용 가능한 구조 확보

프로젝트 개요

신규 고객 확보와 이용 유도를 위한 멤버십 등급제 신규 설계. 최상위 등급은 정원 100명으로 제한된 랭킹제로 운영.

팀 구성 및 역할

  • 단독으로 등급 정책 설계, 승강급 자동화, 등급별 혜택 연동 담당

주요 업무

  • 이용 횟수 기반 자동 승급 로직 설계
  • 정원 100명 제한 최상위 등급의 랭킹제 운영
  • 등급별 포인트 적립 비율 조정과 이용권 지급 등 혜택 연동

문제 해결 과정

  • Redis Sorted Set 정원제
    • 매번 SQL 정렬은 회원 증가에 비례해 부하 ↑
    • Sorted Set으로 이용 실적을 score로 적재, 매월 배치로 한 번에 갱신
    • 상위 100명 유지, 정원 밖 강등, 신규 진입 승급을 자동화
  • 혜택 정책 메타 테이블 분리
    • 등급별 적립 비율과 이용권 지급 정책을 메타 테이블로 분리해 코드 배포 없이 정책 변경 가능

기여 성과

  • 도입 대비 이용률 14% 증가
  • Redis Sorted Set 기반 정원제로 회원 규모와 무관한 일정한 부하 확보
