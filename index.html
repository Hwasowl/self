프로젝트 개요

결제 로직은 모놀리식, 환불과 정산은 백오피스 서버에 흩어져 있어 변경사항이 발생하면 중복 수정해야 하는 구조. 이에 유지보수와 책임 분리를 위해 별도 결제 서비스로 분리

팀 구성 및 역할

  • 3명 백엔드 팀으로 진행
  • 주로 환불 도메인 전반과 PG 연동, 백오피스 연동 작업을 수행

주요 업무

  • Node.js 모놀리식의 모호한 결제 도메인 개념을 정리해 Java 엔티티와 값 객체로 재모델링
  • 모놀리식, 백오피스에 존재하는 결제와 환불 로직을 정리해 결제 시스템에 이관 및 연동
  • 나이스페이먼츠, 카카오페이먼츠, 토스페이먼츠 PG 연동(결제, 환불) 작업
  • 비동기 분리 작업

문제 해결 과정

  • 두 DB 동시 갱신 정합성
    • 대안 비교: 평균 50TPS 규모의 단일 인스턴스라 2PC와 Saga 패턴은 복잡도 대비 이득이 없다고 판단.
    • 유입 트래픽과 운영 난이도를 고려해 ChainedTransactionManager로 두 DB를 순차 커밋 (결제 -> 레거시 순서)
    • 잔여 리스크인 부분 커밋 실패 발생 시 Slack 알림 + 복구 쿼리로 대응
  • 정산 비동기 분리
    • 결제 완료 시점에 SQS로 결제 이벤트 발행, 정산 서버 비동기 컨슘
    • 실패는 DLQ 재시도와 청구서 ID 기반 멱등성 키로 중복 정산 방지
  • 테스트로 도메인 정확성 보장
    • 단위와 통합 테스트 300개 이상 작성, 커버리지 95% 유지

기여 성과

  • 정산 및 로깅 처리 비동기화로 응답 속도 2.1s → 0.4s
  • 모놀리식, 백오피스 서버 무중단 분리 이관

Java 17, Spring Boot, Redis, MySQL 5.7

프로젝트 개요

지역 조회 API의 RDS Full Scan 쿼리가 CPU 사용량을 끌어올려 Timeout이 발생했고, 전체 서비스 장애로 확산. 이에 캐싱 전략을 점진적으로 도입하며 해결.

팀 구성 및 역할

  • 단독으로 원인 분석, 캐싱 전략 수립, 점진적 마이그레이션

주요 업무

  • 운영 구역 폴리곤, 가맹점, 킥보드 위치를 매 화면 진입마다 응답하던 지도 조회 API의 캐싱 설계와 적용
  • 로컬 캐시 → Redis 글로벌 캐시 → 공간 쿼리로 캐싱 전략 점진적 마이그레이션
  • Write Through, TTL 기반 캐시 일관성 관리와 응답 크기 축소로 장애와 렌더링 지연 해소

문제 해결 과정

  • 1단계: 로컬 캐싱
    • 변경 빈도 낮은 지역 데이터에 WAS 로컬 캐싱 적용으로 6s → 0.4s
    • PM 규제 법안이 추가된 이후, 지역 데이터 변경이 잦아지며 WAS 증설 시 인스턴스별 캐시 불일치 발생
  • 2단계: Redis 글로벌 캐싱
    • Write Through + TTL 설정으로 지역 데이터 관리
    • WAS 간 일관성 확보로 캐시 불일치 해소
    • 다만 가맹점 증가로 응답 크기가 커지며 프론트 렌더링 지연 발생
  • 3단계: 공간 쿼리
    • MySQL Geometry 함수로 사용자 인접 지역만 조회하도록 변경
    • 응답 크기 자체를 축소해 렌더링 지연 해소

기여 성과

  • 처리 속도 6s → 0.4s (15배 개선)
  • RDS Full Scan으로 인한 장애 재발 차단

반납지연 CS 처리 효율화

프로세스 개선으로 일 평균 VOC 처리량 56% 개선

Node.js, Express, React, Amplitude, Datadog Logs API

프로젝트 개요

반납 실패 문의 시 CS가 실제 종료 여부와 적정 요금을 알 수 없어 매번 개발자 확인과 수기 계산에 의존하던 흐름을, 종료 시점 추정과 환불 차액 산정까지 한 화면에서 처리하도록 개선

팀 구성 및 역할

  • 단독 진행. 로깅 연동, 실이용 추정 로직, 적정 요금 역산, 응대 화면 통합

주요 업무

  • Amplitude, Datadog 로그와 위치 정지 패턴으로 실이용 종료 시점 추정
  • 지역 요금제로 적정 요금을 역산해 환불 차액을 CS 화면에 제시
  • 반납 이슈 지도 마커 시각화, 중복 로그 호출 캐싱

문제 해결 과정

  • 실이용 종료 시점 추정
    • 종료 기록이 없는 건은 클라이언트와 서버 로그를 대조하고, 킥보드 GPS가 멈춘 위치와 시점으로 실제 반납 시점 추정
    • 반납 시도 위치와 정지 구간을 지도 마커로 표기해 응대자가 추정 근거를 화면에서 확인
  • 적정 요금 역산
    • 추정 종료 시각과 지역 요금제로 적정액을 계산, 부과액과의 차액을 표시해 수기 계산 제거
  • 중복 호출 캐싱
    • 동일 로그 반복 호출을 캐싱으로 묶어 외부 API 호출 감소

기여 성과

  • CS가 개발자 확인 없이 반납 이슈를 직접 처리 → 개발자 확인 병목 제거
  • 적정 요금 역산으로 수기 계산 제거, 환불 정확도 확보
  • 일 평균 VOC 처리 32건 → 50건 (약 56% 증가)
