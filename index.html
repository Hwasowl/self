프로젝트 개요

결산 시즌, 회계 전표 1만 건 이상 일괄 처리 시 최대 90초가 소요되거나 OOM 문제가 발생하며 이를 개선

팀 구성 및 역할

  • 단독 수행 — 병목 진단, 배치 구조 설계, 성능 최적화

주요 업무

  • 건별로 동작하던 전표 승인, 취소 로직의 병목 진단
  • 건수마다 형태가 변하던 MyBatis 동적 쿼리를 고정 SQL + BATCH 실행으로 재설계해 실행계획 재사용 유도

문제 해결 과정

  • 병목 진단
    • 건수만큼 UPDATE 문을 이어붙이는 MyBatis 동적 쿼리라, 건수와 데이터에 따라 SQL 형태가 바뀌어 옵티마이저가 반복적으로 하드 파싱하는 문제 발생
  • 실행계획이 재사용되도록 SQL 실행 구조 변경
    • 건수마다 변하던 동적 UPDATE를 단일 UPDATE 문장으로 변경해, 매 실행 같은 PreparedStatement를 재사용
    • SQL 텍스트가 고정되어 옵티마이저가 소프트 파싱하므로, 건수만큼 반복되던 하드 파싱 문제 개선
  • MyBatis Executor 충돌 → 트랜잭션 경계 분리
    • 상위 서비스의 조회가 SIMPLE executor로 먼저 실행돼 같은 트랜잭션에서 BATCH 전환이 막히는 문제를, 트랜잭션 전파 옵션 REQUIRES_NEW로 분리해 해결
    • SAVEPOINT 부분 롤백으로 실패한 전표만 스킵하고 나머지는 처리를 지속, 실패 전표번호는 따로 수집해 안내
  • BATCH 세션 flush로 메모리 제어
    • 1,000건마다 세션에 쌓인 누적분을 비워 힙 상한을 유지하고, 청크 크기를 조정하며 균형점 확인

기여 성과

  • 1만 건 이상 승인 처리 90초 → 2초 (약 45배)
  • 결산 피크의 OOM 실패 차단, 실패 건만 격리해 부분 실패가 전체 배치를 막지 않도록 운영 안정성 확보

프로젝트 개요

모놀리식과 백오피스에 흩어져 변경마다 중복 수정하던 결제/환불/정산 로직을 Node.js에서 Spring Boot 기반 결제 서비스로 마이그레이션

팀 구성 및 역할

  • 3명 백엔드 팀으로 진행
  • 주로 환불 도메인 전반과 PG 연동, 백오피스 연동 작업을 수행

주요 업무

  • Node.js 모놀리식에 흩어져 있던 결제 도메인 로직을 Java 엔티티와 값 객체로 재설계
  • 모놀리식, 백오피스에 존재하는 결제와 환불 로직을 정리해 결제 시스템에 이관 및 연동
  • 나이스페이먼츠, 카카오페이먼츠, 토스페이먼츠 PG 연동(결제, 환불) 작업
  • 정산과 로깅을 SQS 기반 비동기로 분리

문제 해결 과정

  • 레거시 정합성 관리
    • 결제 도메인은 분리했지만 레거시가 아직 결제 데이터를 읽어, 결제 시 신규 결제 DB와 레거시 DB에 함께 기록해야 했음
    • 두 DB를 원자적으로 묶는 2PC까지는 규모상 과하다 판단, 순서대로 커밋하는 ChainedTransactionManager 도입
    • 순차 커밋은 실패 시 부분 반영 문제가 발생할 수 있어, 그 경우 Slack 알림과 복구 쿼리로 보정
  • 정산 비동기 분리
    • 결제/환불 커밋 이후, SNS 토픽에 이벤트 발행 → 정산 서버가 구독 SQS로 컨슘
    • 청구서 ID 기반 멱등성 키로 중복 정산을 방지하고, 처리 실패 시 DLQ를 통한 관리
  • 테스트로 도메인 정확성 보장
    • 단위와 통합 테스트 300개 이상 작성, 커버리지 95% 유지

기여 성과

  • 정산 및 로깅 처리 비동기화로 응답 속도 2.1s → 0.4s
  • 레거시와 신규 서비스가 공존하는 상태에서 무중단 분리 이관 완료

프로젝트 개요

지도 조회 API가 RDS CPU를 포화시켜 Timeout이 발생하고, 공유 DB를 쓰는 전체 서비스 장애로 확산. 캐싱으로 당장의 부하를 걷어낸 뒤, 최종적으로 조회 범위 자체를 줄여 원인을 제거

팀 구성 및 역할

  • 단독으로 원인 분석, 캐싱 전략 수립, 점진적 마이그레이션

주요 업무

  • 운영 구역 폴리곤, 가맹점, 킥보드 위치를 매 화면 진입마다 응답하던 지도 조회 API의 캐싱 설계와 적용
  • 로컬 캐시 → Redis 글로벌 캐시 → 공간 쿼리로 캐싱 전략 점진적 마이그레이션
  • Write Through, TTL 기반 캐시 일관성 관리와 응답 크기 축소로 장애와 렌더링 지연 해소

문제 원인

  • 지도 화면에 진입할 때마다 운영 구역 폴리곤을 사용자 위치와 무관하게 모두 반환해 매 요청이 Full Scan으로 실행됨
  • 서비스 지역과 가맹점이 늘수록 조회량과 응답 크기가 함께 커졌고, 피크 시간대 RDS CPU가 포화되며 같은 DB를 쓰는 전체 서비스로 장애가 확산됨

문제 해결 과정

  • 1단계: 로컬 캐싱
    • 변경 빈도 낮은 지역 데이터에 WAS 로컬 캐싱 적용으로 6s → 0.4s
    • 하지만 PM 규제 법안이 추가된 이후 지역 데이터 변경이 잦아지며, WAS 증설 시 인스턴스별 캐시 불일치 발생
  • 2단계: Redis 글로벌 캐싱
    • Write Through + TTL 설정으로 지역 데이터 관리
    • TTL 만료 시 동시 요청이 DB로 몰리는 캐시 스탬피드를, 분산 락으로 잡으며 방지
    • WAS 간 일관성 확보로 캐시 불일치 해소
    • 다만 지속적인 가맹점 증가로 응답 크기가 커지며 프론트 렌더링 지연 발생
  • 3단계: 공간 쿼리로 조회 범위 축소
    • MySQL Geometry 함수로 전 지역 대신 사용자 인접 지역만 조회하도록 변경, Full Scan의 원인이던 조회 범위 축소
    • 응답 크기도 함께 줄어 프론트 렌더링 지연 해소

기여 성과

  • 처리 속도 6s → 0.4s (15배 개선)
  • 조회 범위 축소로 Full Scan 원인을 제거, RDS 사용량 56% 절감으로 장애 재발 차단
